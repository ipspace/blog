    If you tested and it worked this is no way an indication that it is supported.

    Read documentation, especially Restrictions and Limitations sections, before putting anything in production deployment. Google is very efficient way to navigate vendors documentation that spans multiple unconnected or unlinked documents. If can't find an answer - open a ticket with your vendor to clarify specify SW/HW combo features/limitations.

    The prime abusement is 'ip nat enable"; were folks use it as a replacement for 'ip nat inside/outside'. If both are working in the global routing table does not mean that both are supported. So, when something fails in production, after 'about 15 seconds' of testing in the lab, then network redesign might be the only option.

    2. lcw 05 December 2017 17:36
      A decade ago, I challenged my coworkers to come up with an IOS config for transparent interception, so that HTTP from any source to any destination is "proxied" to one specific host. With IOS, this means applying both client source PAT and server destination NAT (rotary pool) to the same flow, a tricky feat since those functions have inverted ideas about NAT "inside" and "outside". I was expecting a solution that did each NAT function in its own VRF, with inter-VRF traffic being manhandled over a crossover cable.

      But one clever fellow combined "ip nat enable" for the client source PAT and "ip nat inside|outside" for the server destination NAT on the same pair of interfaces. It was an elegant, single hop, single VRF solution that was completely consistent with the NAT order-of-operation and, as far as I'm concerned, is fully supported by TAC!
